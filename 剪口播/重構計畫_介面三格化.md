```
① 你只做 2 件事（橘）          ② 機器一鍵全包（虛線框，你不碰）          ③ 審核介面定稿（橘）
─────────────────             ──────────────────────────────             ─────────────────
丟 OBS 資料夾                  抽音 → 自動抽熱詞 → BytePlus 辨識+字級時間戳    掃 3–5 分鐘
(可選) 給參考文檔               → 偵測口誤 → 落刀分層出預剪                   攔誤殺、微調
                                                                          AI 改過/沒把握處高亮

外加兩件系統姿態變更：
- **退役 F1 / 訓練層**：停掉統計式自我優化，剪片主線保留
- **回饋迴路**：使用者口頭回報常犯錯 → 助理翻成機器規則檔 → 下次自動套用

## 關鍵設計決策（已拍板）

1. **安全高亮做彈性版**：有參考文檔 → 辨識稿 vs 文檔對比，差異處高亮（防「說 a 變 b」）；沒文檔 → 退回基本高亮（AI 標刪處 + 幻覺詞）。
2. **退役 F1 = 只退 L1（F1 煙霧測試）與自動優化**；**保留 L2（輸出 diff，無 AI 成本）、L4（成品驗證，無 AI 成本）當安全網**。這兩個不是 F1，是結構性防呆，留著零成本。
3. **回饋迴路靠規則檔**：機器一鍵跑時助理不在現場，所以「下次不犯」必須落到機器讀得到的檔（`用户习惯/*.md`、熱詞詞典、保護詞、`hallucination_blacklist.json`）。助理負責把白話翻成這些檔。
4. **參考文檔一檔兩用**：辨識前抽熱詞（防錯）+ 辨識後比對高亮（抓漏）。
5. **訓練資料不刪**：`user_corrections.jsonl`、training_output 全留，當資產。

## 分階段

### Phase 0 — 退役 F1 / 訓練層（低風險，先做）
- training_server.js：隱藏「開始訓練 / 自動優化」UI 入口（不刪程式，只下架按鈕）
- 更新 `testing.md`、`rules/01-test-before-modify.md`、CLAUDE.md：改碼門檻從「L1 F1 ≥ 96%」改成「L2 輸出 diff = 0 + L4 成品驗證」
- 保留 user_corrections.jsonl 持續寫入
- 影響檔：training_server.js、testing.md、rules/、CLAUDE.md

### Phase 1 — 回饋迴路骨架（隨時可開，持續性）
- 盤點機器讀的規則檔，整理成一張「助理改哪裡」對照表
- 新增 `用户习惯/常犯錯誤.md`（或 `口誤詞庫`）：累積使用者回報的具體錯字對（a→b）、口頭禪、固定要刪/要留
- 確保剪輯匯出時 user_corrections.jsonl 持續記錄（即使退役訓練）
- 流程文件化：使用者回報 → 助理更新哪個檔 → 下次生效
- 影響檔：用户习惯/（整理+新增）、文件

### Phase 2 — 自動抽熱詞 ❌ 原設計作廢（2026-06-30 實測）
**實測結論**：BytePlus 海外站 `request.corpus.context`（行內熱詞）**會讓辨識失敗（55000001）**，submit 接受該欄位但 query 掛掉。唯一可用的是 `boosting_table_name/id`——**預先上傳的持久詞表**，不是逐片動態。所以「每片從 reference.txt 即時抽熱詞」這條路死了。
另：jieba 對繁體斷詞品質差（人名被切錯），即使要做詞表也不適合。

**改走兩條（取代原 Phase 2）**：
- **2a（可選、之後再說）持久 boosting 詞表**：把你頻道常出現的專名（書名/作者/概念）做成一份詞表，上傳一次、每片引用。需先摸清上傳詞表的 console/API 流程。
- **2b（併入 Phase 4）參考文檔改用在「辨識後」**：不在辨識當下注入，而是辨識完用 reference.txt 比對校正/高亮 → 這本來就是 Phase 4，且正中「說 a 變 b」安全目標。

→ **參考文檔的價值整個移到 Phase 4**（辨識後），不再依賴 ASR 熱詞。

### Phase 3 — 一鍵打包 ①→②（體驗收口）
- 介面：丟 OBS 資料夾（+可選 reference.txt）→ 一顆按鈕 → 跑完整 ② 鏈、進度條
- 現況 startCutProcess 已串大部分；要做的是 UI 收口 + 把熱詞步驟插進流程
- 影響檔：training_server.js（或獨立精簡剪片 UI）

### Phase 4 — ③ 審核介面安全高亮（你最在意的防呆）
- 新腳本：reference-diff 對齊（類似 align_corrected.js，但輸出「辨識稿 vs 文檔」差異 flag）
- 審核 UI（review_server.js / review.html）加高亮層：
  - 有文檔：差異處高亮（防 a→b）
  - 沒文檔：退回基本（AI 標刪 + 幻覺詞）
  - AI 改過處一律高亮
- 影響檔：review_server.js、review.html、新對齊腳本

## 待研究 / 風險

- **BytePlus 熱詞支援**（Phase 2 命脈）：新 bigmodel 端點到底吃不吃 hotwords，要先打 API 驗證再決定 Phase 2 怎麼做。

- **既有 F1 / L2 fixture 已半失效**：換引擎後字流變了，舊黃金答案索引對不回。退役後不重建即可；若哪天想恢復 L2，要用新引擎重做 fixture。
